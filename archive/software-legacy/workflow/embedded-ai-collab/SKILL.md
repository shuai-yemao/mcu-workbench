# 嵌入式 AI 协作编程

> 与 AI 协作进行嵌入式 C 开发的标准流程。
> 自动加载编码规范、Prompt 模板和审查清单，确保 AI 生成的代码符合项目标准。

## 触发条件

当用户表达以下意图时，**必须调用本技能**：
- "帮我写 {外设} 驱动"
- "生成 {模块} 的 BSP 代码"
- "用 AI 开发 STM32 代码"
- "嵌入式 AI 协作"
- 任何涉及 AI 生成嵌入式 C 代码的请求

## 启动流程

<HARD-GATE>
在完成 Phase 0（加载规范 + 明确需求）之前，**不得生成任何代码**。
即使用户说"先写一个看看"，也必须先加载规范。
</HARD-GATE>

### Phase 0：加载规范 + 明确需求

**步骤 0.1 — 加载约束技能**

首先依次加载以下技能到当前上下文：

1. 调用 `embedded-ai-coding-standard` — 加载编码规范
2. 调用 `embedded-ai-code-review` — 加载审查清单

（Prompt 模板按需加载 `embedded-ai-prompt-templates`）

**步骤 0.2 — 收集需求信息**

向用户确认以下信息（缺一不可）：

| 信息 | 示例 | 用途 |
|------|------|------|
| 目标外设/模块 | MPU6050, AHT21, W25Q64 | 确定模块名和功能 |
| 通信接口 | I2C, SPI, UART | 确定底层 HAL |
| 设备地址 | 0x68 (7-bit) | 通信参数 |
| MCU 型号 | STM32F411CEU6 | HAL 库选择 |
| 需要的函数 | read_reg, write_reg, init... | 确定接口 |
| 硬件约束 | 400kHz, 4.7kΩ 上拉 | 安全参数 |

**步骤 0.3 — 确认开发模式**

```
请选择开发模式：

A. 逐函数模式（推荐）— 一次生成一个函数，编译通过后再继续
B. 模块模式 — 一次生成完整模块（适合简单/标准外设）
C. 审查模式 — 审查现有代码是否符合规范
```

### Phase 1：架构规划

明确分层结构和模块边界：

```
确认分层：
┌─────────────────┐
│ App 层           │ ← 业务逻辑（{具体功能}）
├─────────────────┤
│ Driver 层        │ ← 数据解析（{具体功能}）
├─────────────────┤
│ BSP 层           │ ← 寄存器操作（{具体函数列表}）
├─────────────────┤
│ HAL 层（不修改）  │ ← STM32 HAL 库
└─────────────────┘

确认模块依赖：
- bsp_{模块} → adapter_{接口} → HAL_{外设}
- driver_{模块} → bsp_{模块}
```

输出架构图，等待用户确认。

### Phase 2：接口设计

在写任何实现之前，先定义接口头文件：

```
生成头文件 ec_bsp_{模块}_driver.h，包含：
- 宏定义（寄存器地址、位掩码）
- 枚举（状态、量程、模式）
- 结构体（配置、数据）
- 函数声明（带完整 doxygen 注释）
```

**检查点**：确认头文件符合 `embedded-ai-coding-standard` 模板。

### Phase 3：AI 生成（逐函数）

**核心规则：每次只生成一个函数。**

```
循环执行：
1. 用户/Prompt模板 描述下一个函数的需求
2. AI 生成函数实现（头文件声明 + 源文件实现）
3. 对照 embedded-ai-code-review 逐层审查
4. 用户编译验证
5. 通过 → 继续下一个函数
   失败 → 修复 → 重新验证
```

每生成一个函数后，先自检（对照 `embedded-ai-coding-standard` 的输出前自检清单），再交给用户编译。

### Phase 4：代码审查

整个模块生成完毕后，执行完整审查：

```
对照 embedded-ai-code-review 四层清单：
□ 第一层：语法检查（编译 -Wall -Wextra）
□ 第二层：规范检查（命名、缩进、注释格式）
□ 第三层：逻辑检查（判空、返回值、越层调用）
□ 第四层：安全检查（中断、DMA、时钟、看门狗）
□ 反模式检查（寄存器地址、硬件约束）
```

输出结构化审查报告（参考 `embedded-ai-code-review` 审查输出模板）。

### Phase 5：集成验证

```
1. 编译整个模块 → 零警告零错误
2. 链接到项目 → 无符号冲突
3. 单元测试（如适用）→ 通过
4. 硬件验证 → 寄存器读写正确、数据符合预期
```

## 快速参考：各阶段输出物

| 阶段 | 输出 | 审查关口 |
|------|------|---------|
| 0. 加载规范 | 规范已加载到上下文 | — |
| 1. 架构规划 | 分层图 + 依赖关系 | 用户确认 |
| 2. 接口设计 | 头文件 `.h` | 对照 header-template |
| 3. AI 生成 | 逐函数 `.c` 实现 | 每函数自检 + 编译 |
| 4. 代码审查 | 审查报告 | 四层清单 |
| 5. 集成验证 | 编译通过 + 硬件验证 | 零警告 + 功能正确 |

## 与 ECC 工作流集成

本技能可与 ECC agent 流水线配合使用：

```
embedded-ai-collab (本技能)
    │
    ├─ Phase 1-2 → planner agent（架构规划）
    ├─ Phase 3   → 人工 + AI 逐函数生成
    ├─ Phase 4   → code-reviewer agent（代码审查）
    │               security-reviewer agent（安全检查）
    └─ Phase 5   → build-error-resolver agent（编译修复）
```

如需完整的自动化编排，可触发 `embedded-ai-collab` 工作流（详见 `.claude/workflows/embedded-ai-collab/`）。
