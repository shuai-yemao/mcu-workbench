# BSP 外部器件适配

## 负责什么

本 skill 负责“找到或评估外部器件驱动，并让它符合目标工程的 BSP 边界”。输出是可审查的适配方案、许可证结论、变更清单和集成步骤。

它不负责直接编写平台无关的设备协议状态机（使用 `bsp-device-driver`），不负责多实例服务和线程（使用 `bsp-device-service`），也不负责 CubeMX/HAL 的初始化配置（使用 `platform-stm32-hal`）。

## 必要输入

- 器件型号、数据手册或已知协议。
- 总线类型与实例，例如 I2C/SPI/UART 及其目标工程接口。
- 目标工程 BSP 分层约定、当前代码路径和允许的许可证策略。
- 可选：候选开源仓库或已有 C/H 源码。

## 受控适配流程

1. **确认边界**：先读取项目的 `AGENTS.md`、现有 BSP API、总线抽象和构建方式；不得假定目录结构或 HAL handle 名称。
2. **搜索与许可证筛选**：优先选维护活跃、许可证明确且与商业使用兼容的实现。记录仓库、版本/提交、许可证和不采用候选的原因；GPL 或无许可证代码不得直接合入。
3. **审查接口**：列出公共 API、传输函数、延时/锁依赖、错误语义和全局状态。识别哪些部分是器件协议，哪些部分是平台耦合。
4. **最小化改动**：只替换通信、延时、日志和构建集成层；不要改动经过验证的器件算法，除非数据手册或测试证明其错误。
5. **建立 BSP 契约**：把外部依赖改为可注入接口，公开语义化设备 API。若需要完整生命周期、错误回滚或 Fake/Mock，交给 `bsp-device-driver`。
6. **验证**：先对编译和静态检查，再用目标板确认设备探测、正常读写、超时/断线和恢复路径。构建问题交给相应的 `tool-build-*` skill。

## 变更清单

- 保留上游来源、版本和许可证信息。
- 记录新增/修改文件、公开 API、需要的工程组与 include path。
- 不覆盖用户已有 BSP 文件；遇到同名组件时生成迁移建议而不是自动替换。
- 不包含用户机器绝对路径、私有脚本位置或不可复现流水线。

## 交接

| 需求 | 使用的 skill |
|---|---|
| 设计单个设备的协议、状态和生命周期 | `bsp-device-driver` |
| 管理多个设备实例、队列、线程与事件 | `bsp-device-service` |
| 将注入接口绑定到 HAL/RTOS | `bsp-platform-adapter` |
| 配置 STM32 外设、CubeMX、HAL | `platform-stm32-hal` |
| 编写 Fake/Mock 或板端 Unity 测试 | `quality-unity-testing` |
