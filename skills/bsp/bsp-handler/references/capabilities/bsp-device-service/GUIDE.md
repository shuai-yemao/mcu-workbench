# BSP 外设 Service

## 工作边界

把多个 Driver 实例组织成上层可用的服务。旧名称中的 “Handler” 指同一职责；统一命名后使用 Service，强调它管理实例、调度和资源，不重复实现设备协议，也不绕过 Driver 访问硬件。

先读取目标工程的 Handler 头文件、实现文件、Driver 公共接口和 RTOS/Adapter 接口。AHT21 的多实例模式见 [references/aht21-handler-pattern.md](references/aht21-handler-pattern.md)。

## 执行流程

1. **划分职责**
   - 列出 Handler 对外 API、Handler 状态码、Driver 状态码和它们的映射关系。
   - 标出 Handler 拥有的实例数组、队列、线程、事件和回调资源。
   - 明确 Driver 实例由谁创建、谁销毁、Handler 是否拥有其存储空间。

2. **设计实例管理**
   - 设定最大实例数和实例索引/句柄规则，拒绝越界和重复注册。
   - 规定初始化、启动、停止、删除和重新初始化的合法状态转移。
   - 初始化失败时按逆序释放已创建的队列、线程、事件和 Driver 资源。
   - 所有公共 API 先检查 Handler 指针、句柄、实例状态和输出指针。

3. **接入调度资源**
   - 将延时、队列、线程、临界区和事件回调通过 OS Adapter 注入。
   - 明确线程入口、队列元素大小、超时单位和回调执行上下文。
   - 不在中断上下文中调用可能阻塞的 Driver 或普通队列 API。
   - 记录并返回每个 OS 操作的失败状态，不把失败伪装成“无数据”。

4. **处理数据流**
   - Handler 调用 Driver 的公开读数接口，验证结果后再向上层发布。
   - 保持温度、湿度等数据的单位、时间戳、有效位和错误来源。
   - 回调失败、队列满、读数超时和 Driver 错误要有明确策略：丢弃、重试、告警或停止。
   - 不让一个实例的错误污染其他实例的状态。

5. **生成测试矩阵**
   - 正常初始化、单实例、多实例、最大实例数和删除后重建。
   - `NULL` 参数、越界索引、重复注册、未初始化调用和重复销毁。
   - Driver 读数失败、队列创建/发送/接收失败、线程创建/删除失败。
   - 回调触发、回调为空、队列超时、临界区失败和资源回滚。
   - 验证每条失败路径的资源调用次数、Handler 状态和 Driver 状态映射。

6. **验证集成**
   - 优先用 Fake Driver 与 Fake OS Adapter 做主机或目标板单元测试。
   - 再做 Keil 测试构建和正常固件构建，确认测试入口未泄漏到生产 `main()`。
   - RTT 日志至少包含 Handler 创建、资源失败、Driver 错误和测试汇总。

## 设计禁区

- 不在 Handler 中复制 I2C/SPI 命令和设备寄存器协议。
- 不使用裸的全局队列、线程或传感器数组隐藏资源所有权。
- 不把 Driver 错误直接当作 Handler 成功，也不丢失原始错误来源。
- 不在资源尚未完整创建时启动后台线程。
- 不为了让测试通过而跳过真实的回滚和销毁逻辑。

## 交付检查

- Handler 与 Driver 的依赖方向清晰，公共 API 能说明所有权和调用时机。
- 最大实例数、重复调用和资源失败都有确定返回值。
- 线程、队列、事件和 Driver 资源在失败与正常路径都能回收。
- 测试覆盖多实例、边界、依赖失败、异步路径和错误映射。
