# 项目学习与笔记生成

## 用途

本 Skill 把一个真实嵌入式项目转化为可复习的 Obsidian 学习笔记：先读取代码和已有笔记，再按章节逐步提问，确认用户能解释关键机制，最后生成带代码证据、Mermaid 图和 Q&A 的 Markdown 笔记。

它属于工具方向，负责“提问、整理、记录和复核”，不替代软件架构设计或具体驱动实现。

## 输入

开始前确认以下信息：

1. 项目根目录或模块/文件路径；
2. 学习目标，例如 Driver、BSP Handler、OS Task、LVGL 页面或一条数据链路；
3. Obsidian Vault 路径和目标笔记目录；
4. 是否更新已有笔记，以及是否允许最终写入文件。

如果路径、分支、构建配置或硬件事实不明确，先列出缺失证据，不凭经验补齐。

## 工作流

### 1. 扫描项目

- 读取目标模块相关的 `.h`、`.c`、构建文件、配置文件和启动调用链；
- 建立“文件 → 类型/函数 → 调用者 → 下层接口”的证据表；
- 搜索同名或相关 Obsidian 笔记；
- 对比代码与笔记，标出遗漏、过时内容、未使用接口和未验证假设；
- 先把扫描结果交给用户确认，再开始提问。

### 2. 逐节提问

每次只问一个问题，等待用户回答后再继续。推荐顺序：

1. 一句话定义：这个模块负责什么；
2. 实际意义：解决什么痛点；
3. 应用场景：在哪些路径被使用；
4. 核心机制：触发者、数据流、消费者和边界条件；
5. 关键参数：默认值、公式和选择理由；
6. 实际操作：初始化、调用、错误处理和释放顺序；
7. 常见问题：现象、根因、修复和验证；
8. 分级 Q&A：基础、进阶、困难各 1–2 题。

“一句话说清”不是复述函数名，而是同时说明职责、机制和设计意图。回答不完整时，先给不超过 100 字的纠正，再追问一个确认问题；用户说“不知道”或“跳过”时记录薄弱点并继续，不强行追问。

### 3. 生成笔记

完成教学和 Q&A 后生成 Markdown 草稿，至少包含：

- 一句话引言和模块职责；
- 文件、结构体、公开 API 和调用链证据；
- 核心逻辑的架构图或时序图；
- 与项目一致的关键代码片段，并带 `文件路径:行号`；
- 参数/公式、实际操作步骤和常见问题；
- 用户回答形成的 Q&A；
- 未验证项、薄弱点和后续实验；
- 代码文件与相关 Obsidian 笔记的 WikiLink。

推荐文件名：`[芯片或模块]的[文件角色]文件架构设计思路.md`。

### 4. 审查和写入

写入前展示完整草稿和目标绝对路径，等待用户确认。确认后再写入 Vault。写入后重新检查：

- 公开 API、结构体和配置宏是否覆盖；
- 初始化/去初始化顺序是否正确；
- 图表、代码行号和 WikiLink 是否有效；
- 推断内容是否明确标记为推断；
- 代码已变更时是否提示笔记可能过时。

## 输出格式

每轮交互输出当前问题或当前章节，不提前生成未经确认的完整答案。最终交付包含：

```text
项目/模块范围
证据文件与调用链
逐题理解结果与薄弱点
Obsidian Markdown 草稿
写入路径与用户确认状态
未验证项和下一步
```

## 边界与交接

- 跨层架构、目录映射和工程审计交给 `workflow-project-integration`；
- APP、OS、BSP、Core、Driver 或 Middleware 的具体设计分别交给对应 canonical Skill；
- Obsidian 文件写入必须经过用户确认；
- 不把代码推断写成已验证的硬件事实；
- 不在本 Skill 内直接修改业务代码、驱动代码或架构目录。
