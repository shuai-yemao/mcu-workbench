# BSP 外设 Driver

## 工作边界

把一个具体外设封装成与 MCU 平台解耦的 Driver。Driver 只处理设备协议、设备状态和公开行为；GPIO、I2C、SPI、时间、RTOS 和中断由注入的接口提供。

先读取目标工程的 `AGENTS.md`、Driver 头文件、实现文件、配置头文件和现有 Adapter。AHT21 的接口提炼见 [references/aht21-driver-pattern.md](references/aht21-driver-pattern.md)，它是实例，不是固定 API。

## 执行流程

1. **提取契约**
   - 列出公共构造/实例化函数、实例方法、参数、返回状态和输出数据单位。
   - 标出实例状态字段、私有资源、底层依赖和可重入要求。
   - 区分“设备无响应”“参数错误”“超时”“资源失败”和“调用时机错误”。

2. **设计依赖注入**
   - 为通信、时间基准、延时、RTOS 让出、锁和中断保护定义函数指针表。
   - Driver 不包含 STM32 HAL 具体头文件，不直接访问 GPIO/RTOS 全局对象。
   - 让每个依赖可被 Fake/Mock 替换，并能记录调用次数、参数和故障点。

3. **实现生命周期**
   - 实例化前验证顶层指针、依赖结构体和每一个依赖函数指针。
   - 按“绑定依赖 → 初始化总线 → 探测/读取状态 → 完成校准或配置”的顺序实现。
   - 任一步失败都释放已经获得的资源，实例回到可重试的未初始化状态。
   - 对重复实例化、未初始化调用、反初始化和销毁保持明确且稳定的返回值。

4. **实现设备行为**
   - 将协议命令、等待时间、状态位和数据换算集中在 Driver 内部。
   - 对温度、湿度等输出先验证指针和数据帧，再写入输出参数。
   - 睡眠/唤醒等改变设备状态的操作必须更新软件状态或明确说明不更新的原因。
   - 日志只记录诊断所需信息；禁止在时序敏感的 bit-bang 通信循环中打印日志。

5. **生成验证矩阵**
   - 正常实例化和正常读数。
   - 顶层参数、嵌套依赖和输出指针为 `NULL`。
   - 初始化、状态读取、通信、等待和校准失败。
   - 重复实例化、未初始化访问、超时、边界数据、睡眠/唤醒和销毁。
   - 每个案例写出前置状态、Mock 行为、预期状态码、状态变化和资源调用次数。

6. **验证工程集成**
   - 先做静态检查和主机/Fake 测试，再执行 Keil 构建。
   - 检查 Driver 的 include path、源文件分组和宏配置是否真实生效。
   - 目标板验证必须记录初始化、通信错误、返回值和传感器输出；不能只依据“能编译”。

## 设计禁区

- 不把平台 HAL 调用塞进 Driver 以省略 Adapter。
- 不用全局可变状态替代实例状态，除非协议明确要求且文档记录原因。
- 不在错误路径静默吞掉底层状态码。
- 不为了测试暴露私有函数；优先通过公开 API 和注入接口测试。
- 不把某一个 AHT21 的地址、命令或状态位直接复制到其他器件。

## 交付检查

- 公共头文件能说明调用顺序、所有权和返回值。
- 依赖接口可被 Mock 完整替换。
- 失败路径可重试且不泄漏资源。
- Driver 测试覆盖正常、参数、依赖、状态和边界行为。
- 生产构建与测试构建的宏、入口和日志路径没有冲突。
