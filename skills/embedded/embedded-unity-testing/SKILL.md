# 嵌入式 Unity 单元测试

## 工作边界

把单元测试设计文档转成可重复运行的 Unity 测试，并在硬件约束下区分主机/Fake 测试与目标板 RTT 验证。测试代码通过公开接口和注入依赖验证行为，不修改被测模块的业务逻辑来迁就测试。

先读取测试设计文档、被测头文件/实现文件、Adapter 接口、Keil 工程和现有测试入口。AHT21 的单文件测试入口与日志模式见 [references/aht21-unity-pattern.md](references/aht21-unity-pattern.md)。

## 执行流程

1. **建立测试矩阵**
   - 从设计文档提取行为、前置状态、输入、依赖行为、期望返回值、输出值和状态变化。
   - 按正常、参数错误、依赖失败、超时、边界、重复调用、回滚和资源释放分类。
   - 对每条用例指定使用真实 Adapter、Fake Adapter 还是目标板实测；不要把三者混为“单元测试”。

2. **设计 Fixture 与 Mock**
   - `setUp()` 重置实例、Mock 返回值、调用计数、数据帧、时间和资源状态。
   - `tearDown()` 检查测试专用资源已释放，避免状态泄漏到下一条用例。
   - Mock 必须模拟公开依赖接口，并记录关键参数、调用顺序和失败注入点。
   - 对重复参数组合使用表驱动测试，测试名仍要能说明行为和预期。

3. **编写测试入口**
   - 正式固件使用 `module_unity_test_run()` 一类入口，由现有任务调用。
   - 只在独立主机 Target 需要时定义测试 `main()`；不要与固件 `main()` 同时进入同一 Keil Target。
   - 使用 `UNITY_SMOKE_TEST` 或工程既有宏控制测试代码和测试日志，关闭宏后生产路径应保持可构建。
   - Driver、Handler 和 Adapter 测试可以合并到一个模块入口，避免每个文件增加重复 runner。

4. **配置日志**
   - Unity 字符输出先按行缓存，再通过工程统一 logger 输出，例如 `elog_info("UNITY", "%s", line)`。
   - RTT 缓冲区和阻塞策略按测试吞吐量配置，必须保证汇总行 `Tests/Failures/OK` 不丢失。
   - 测试失败日志保留测试名、文件/行号、期望值和实际值；不要只输出一个总失败数。

5. **分层验证**
   - 先执行静态检查、主机/Fake 测试和 `git diff --check`。
   - 测试宏打开时通过 Keil 构建并检查 Unity 汇总；宏关闭时再构建正常固件。
   - 目标板验证 Adapter、RTOS、RTT 和真实传感器时序；明确哪些结论不属于纯单元测试。
   - 记录命令、构建错误/警告数、完整 RTT 汇总、失败案例和复测结果。

## 设计禁区

- 不通过修改生产逻辑或降低断言强度来制造 PASS。
- 不让 Unity runner 引入第二个生产 `main()` 或 `__ARM_use_no_argv` 冲突。
- 不把真实传感器偶尔读到有效值当作所有协议路径已覆盖。
- 不在 `setUp()`/`tearDown()` 中隐藏未验证的全局状态重置。
- 不只看最后的 `OK`；必须确认前面的测试数量和失败数量完整输出。

## 交付检查

- 每条测试都能追溯到设计文档和公开行为。
- Mock 覆盖正常、错误、超时、回滚和资源调用顺序。
- 测试构建与正常构建均通过，入口和宏隔离正确。
- RTT/elog 输出完整，包含测试名、汇总和失败诊断。
- 报告明确区分单元测试证据与目标板运行证据。
